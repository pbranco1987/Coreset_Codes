# Contributing Guide

Thank you for your interest in contributing! This guide explains how to set up a development environment, extend the pipeline with new baselines/objectives/constraints, and submit a pull request.

## Quick Development Setup

```bash
git clone <repo-url> Coreset_Codes
cd Coreset_Codes
python -m venv venv
source venv/bin/activate       # or venv\Scripts\activate on Windows
pip install -e ".[dev]"        # editable install with dev dependencies
```

Verify:
```bash
python -c "import coreset_selection; print(coreset_selection.__version__)"
pytest coreset_selection/tests/   # should pass (some optional deps may skip)
```

Key directories:
- `coreset_selection/` — the core Python package (238 files across 10 subpackages).
- `scripts/` — CLI entry points (15 files).
- `docs/` — all markdown documentation.
- `data/` — raw input CSVs (not modified by normal workflows).
- `manuscript/` — LaTeX sources; **not regenerated by code in this repo** (manuscript-writing code was removed per project policy).
- `EXPERIMENTS-.../` — experiment outputs (gitignored; recreated by running the pipeline).

## Branching and Commits

- Work on a feature branch off `main`: `git checkout -b feat/my-change`.
- Commit messages should have an imperative subject line (≤ 72 chars) and a body explaining *why*. Example:
  ```
  Add kernel k-DPP baseline

  Implements exact k-DPP sampling via eigendecomposition for small N,
  plus per-group quota variant. Closes #42.
  ```
- Before opening a PR:
  - Run the tests: `pytest coreset_selection/tests/`.
  - Run a smoke experiment (see [GETTING_STARTED.md § Smallest possible experiment](./docs/GETTING_STARTED.md#smallest-possible-experiment)).
  - Lint/format if configured (`black`, `ruff`, `flake8`).

Pattern to follow: every baseline is a function `baseline_<name>(X, k, ...) -> np.ndarray`.

### 2. Register it in `__init__.py`

Add your functions to `coreset_selection/baselines/__init__.py`:

```python
from .<your_method> import baseline_<your_method>, baseline_<your_method>_quota

__all__ = [
    # ... existing ...
    "baseline_<your_method>",
    "baseline_<your_method>_quota",
]

    # ... existing ...
    "<your_method>": baseline_<your_method>,
    "<your_method>_quota": baseline_<your_method>_quota,
}
```

### 3. Add tests

Create `coreset_selection/tests/test_<your_method>.py`:

```python
import numpy as np
from coreset_selection.baselines import baseline_<your_method>


def test_returns_k_unique_indices():
    X = np.random.randn(100, 10)
    idx = baseline_<your_method>(X, k=20, seed=0)
    assert idx.shape == (20,)
    assert len(set(idx)) == 20
    assert idx.min() >= 0 and idx.max() < 100
```

### 4. Document it

Add an entry to [docs/api/baselines.md](./docs/api/baselines.md) following the format of existing entries.

### 5. Benchmark it

Run the baseline against the existing 8 methods via the championship analysis (see [docs/api/scripts.md § championship](./docs/api/scripts.md#championship)).

## How to Add a New Objective

Objectives are evaluated inside `SpaceObjectiveComputer`. The pattern:

1. **Implement** a stand-alone class or function in `coreset_selection/objectives/<your_objective>.py` that, given features `X`, can evaluate the objective on a boolean mask in `O(k)` or `O(k²)` time. Cache whatever is mask-independent at `__init__`.
2. **Integrate** into `SpaceObjectiveComputer`:
   - Add a construction branch in `coreset_selection/objectives/computer.py:build_space_objective_computer`.
   - Add a `compute_<name>(mask) -> float` method.
   - Update `compute_all(mask, objectives)` to handle the new objective name.
3. **Plumb** the name through `adaptive_tau.py`:
   - Add the objective name to the `--objectives` allowed list (argparse).
   - Update the experiment-name generator in `build_experiment_name` to produce a descriptive folder name for ablations (e.g., `ablation-<yourobj>only-k100-rep0`).
4. **Test** with a small run and inspect `coreset.npz` to confirm the `F` column includes the new objective.
5. **Document** in [docs/api/objectives.md](./docs/api/objectives.md).

## How to Add a New Constraint Mode

Constraint modes are dispatched in `scripts/launchers/adaptive_tau.py:build_constraint_set`.

1. Decide whether the constraint is **soft** (KL-based, tolerance `τ`) or **hard** (exact quota via projector). Soft constraints live in `coreset_selection/constraints/proportionality.py`; hard constraints live in `coreset_selection/geo/projector.py`.
2. Implement the builder (e.g., `build_<name>_constraint(geo, ..., tau=0.02, name="<name>")`) returning a `ProportionalityConstraint`.
3. Re-export from `coreset_selection/constraints/__init__.py`.
4. Add a branch to `build_constraint_set` in `adaptive_tau.py` that wires it into a `ProportionalityConstraintSet`.
5. Update the `--constraint-mode` `choices=` list in the argparse setup of `adaptive_tau.py`.
6. Document in [docs/api/constraints.md](./docs/api/constraints.md).

## How to Add a New Metric

Metrics are computed by `scripts/analysis/evaluate_coresets.py` via `RawSpaceEvaluator`.

1. Implement the metric function in `coreset_selection/evaluation/<your_metric>.py` with the signature `fn(S_idx: np.ndarray, ...) -> Dict[str, float]`.
2. Call it from either:
   - `RawSpaceEvaluator.all_metrics_with_state_stability` (if it depends on Nyström features/KRR), or
   - Stage [3/5] or [4/5] of `evaluate_coresets.py:evaluate_single_coreset` (if it is independent).
3. The keys you return become columns in `metrics.csv`.
4. Update [docs/api/evaluation.md](./docs/api/evaluation.md).

## Documentation Style Guide

### Python docstrings

Use NumPy style:

```python
def my_function(x: int, y: float = 1.0) -> float:
    """One-sentence summary in imperative form.

    Longer description if needed. Explain *why* the function exists and
    when to use it. Reference manuscript equations where relevant.

    Parameters
    ----------
    x : int
        Description.
    y : float, optional
        Description. Default: 1.0.

    Returns
    -------
    float
        Description.

    Raises
    ------
    ValueError
        If `x` is negative.

    Examples
    --------
    >>> my_function(2, 3.0)
    6.0
    """
    ...
```

### Type hints

**Required** on every public function. Use `typing.Optional[T]`, `List[T]`, `Dict[K, V]`, `Tuple[...]`. For numpy arrays, use `np.ndarray` and document the expected shape in the docstring.

### Markdown documentation

- One H1 per file, at the top.
- Cross-references as relative links: `[name](./other.md)`.
- API-reference entries follow the format defined at the top of [docs/api/index.md](./docs/api/index.md).
- Keep line length around 100 characters for readability in source (markdown renderers wrap automatically).

### Comments

- Prefer self-documenting code; use comments to explain *why*, not *what*.
- Target 15–25 % comment density in new modules.
- Mark `TODO`, `FIXME`, `XXX` uniformly (use `TODO` for planned work, `FIXME` for known bugs).

## Pull Request Checklist

Before marking a PR ready for review:

- [ ] Tests pass locally: `pytest coreset_selection/tests/`.
- [ ] New public functions have NumPy-style docstrings and type hints.
- [ ] New public symbols are re-exported from the appropriate `__init__.py`.
- [ ] API reference ([docs/api/](./docs/api/index.md)) is updated.
- [ ] If behaviour changes, [CHANGELOG.md](./docs/CHANGELOG.md) has a new entry under `[Unreleased]`.
- [ ] Smoke experiment runs: `python scripts/launchers/build_caches.py --reps 0 && python scripts/launchers/adaptive_tau.py --k 30 --space vae --constraint-mode popsoft --rep 0 --cache-dir ... --output-dir /tmp/smoke`.
- [ ] No hardcoded absolute paths (`grep -r "~/Coreset_Codes" scripts/ coreset_selection/` returns nothing new).
- [ ] No new regressions in `grep -r '\.tex' scripts/` (the repo does not generate LaTeX).

## Code of Conduct

Be constructive and respectful. Open issues and PRs with empathy for others' time and context. When reviewing, focus on the code, not the person.
